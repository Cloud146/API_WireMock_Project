# API Mock Project

> Проект представляет собой полноценный тестовый фреймворк для гипотетического сервиса "Book Store API".
> Серверная часть эмулируется с помощью **WireMock**.

## 🛠️ Стек технологий

| Технология | Назначение |
|---|---|
| **Java 21** | Язык программирования |
| **Maven** | Система сборки и управления зависимостями |
| **TestNG** | Фреймворк для запуска тестов и управления их выполнением |
| **RestAssured** | Основная библиотека для написания API-тестов |
| **WireMock** | Инструмент для мокирования (эмуляции) API-сервера |
| **Allure Framework** | Система для построения наглядных и подробных отчетов о тестировании |
| **AssertJ** | Библиотека для написания гибких и читаемых проверок (assertions) |
| **Lombok** | Утилита для сокращения шаблонного кода (boilerplate) в POJO-классах |

## Архитектура проекта

1.  **WireMock для эмуляции API**:
    *   Все ответы сервера описаны декларативно в виде `.json` файлов в директории `src/test/resources/wiremock`.
    *   Используются `mappings` для правил "запрос-ответ" и `__files` для хранения тел ответов.
    *   Применяются продвинутые возможности WireMock, такие как `priority` для управления приоритетом правил и `bodyPatterns` для проверки содержимого тела запроса (`POST`, `PUT`).

2.  **API Client Wrapper**:
    *   Вся логика взаимодействия с API инкапсулирована в классе `BookApiClient`.
    *   Тесты вызывают простые и понятные методы (`bookApiClient.getBookById(1)`).

3.  **Assert Object**:
    *   Вся логика проверки ответов вынесена в отдельный класс `BookApiResponseAssert`, построенный на базе `AssertJ`.
    *   Ассерты предоставляют подробные и понятные сообщения об ошибках в случае падения тестов.

4.  **POJO**:
    *   Для моделирования данных (книги, ошибки) используются POJO-классы. Это позволяет работать с типизированными объектами вместо "сырого" JSON.
    *   `Lombok` используется для автоматической генерации геттеров, сеттеров, конструкторов и других методов.

## Реализованные тесты

Фреймворк покрывает полный набор CRUD-операций, включая позитивные и негативные сценарии.

### Позитивные сценарии (Happy Path)
| ID | Описание | Endpoint | Метод | Ожидаемый Код |
|:---|:---|:---|:---|:---:|
| 1 | Успешное получение книги по ID | `/api/books/{id}` | `GET` | `200` |
| 2 | Успешное создание новой книги | `/api/books` | `POST` | `201` |
| 3 | Успешное обновление существующей книги | `/api/books/{id}` | `PUT` | `200` |
| 4 | Успешное удаление книги по ID | `/api/books/{id}` | `DELETE`| `204` |

### Негативные сценарии (Error Handling)
| ID | Описание | Endpoint | Метод | Ожидаемый Код |
|:---|:---|:---|:---|:---:|
| 5 | Ошибка при получении книги по несуществующему ID | `/api/books/{id}` | `GET` | `404` |
| 6 | Ошибка при создании книги без обязательного поля `title` | `/api/books` | `POST` | `400` |
| 7 | Ошибка при создании уже существующей книги (конфликт) | `/api/books` | `POST` | `409` |
| 8 | Ошибка при удалении несуществующей книги | `/api/books/{id}` | `DELETE`| `404` |
| 9 | Ошибка при обновлении несуществующей книги | `/api/books/{id}` | `PUT` | `404` |
| 10| Ошибка сервера при получении списка книг | `/api/books` | `GET` | `503` |


## Запуск и использование

### Требования
*   Java (JDK) 21 или выше
*   Apache Maven 3.6+

### Запуск тестов

1.  Клонируйте репозиторий:
    ```bash
    git clone https://github.com/[ваш-логин]/[имя-репозитория].git
    ```
2.  Перейдите в корневую директорию проекта.
3.  Выполните команду в терминале:
    ```bash
    mvn clean test
    ```
    Эта команда очистит предыдущие сборки, запустит все тесты и сгенерирует "сырые" данные для отчета в директорию `target/allure-results`.

## Просмотр отчета Allure

После завершения тестов:
```bash
mvn allure:serve
```
